# お気に入り仕様

## 目的

お気に入りファイルの登録、そして管理を楽にするために仮想お気に入りフォルダの追加、閲覧、整理、削除を行えるようにする

## 実装方針

- お気に入りファイルはディレクトリ単位のマーカー管理
- お気に入りフォルダは中央DBでの管理とする

### データ永続化アーキテクチャ

- 新規DBファイルは作らず、既存の`nekoviewer_spread.redb`（[spread_state.rs](../../src/spread_state.rs)、exe直下、見開き状態保存に使用中）に**テーブルを追加**する形で統合する。ファイル名・既存の`spread_state`テーブルはそのまま変更しない（既存ユーザーの状態ファイルとの互換維持のため）
  - 追加テーブル1: `favorite_folders`（フォルダ定義: id/name/marker/color/order。上限200件・名前上限200文字・重複チェック）
  - 追加テーブル2: `favorite_membership`（複合キー：正規化dirパス+filename → 所属folder_idの集合）
- 見開き状態のレコード（`(page_mode, spread_offset)`）自体は拡張しない。見開き状態は「保存操作をした一部のファイルのみ」に存在するのに対し、お気に入りは「任意のファイル」が対象になりうるため、同じレコードに混ぜると見開き用のGCロジック（`gc_dir`）がお気に入り専用データを誤って掃除するリスクが出る。関心事ごとにテーブルを分離する
- `neko_dir.rs`方式（ディレクトリ毎の専用cache.redb）は不採用。あちらはサムネイルという大きいバイナリデータのディレクトリ単位GCのための設計であり、お気に入りのような軽量メタデータをディレクトリ単位で分散させるとDBファイル数だけ増えオーバーヘッドになる
- マイグレーションは不要。redbは書き込みトランザクションで`open_table()`すればテーブルが無ければ自動作成される。読み取り側は「テーブルが存在しない＝空」としてフォールバックする（既存コード`spread_state.rs`/`neko_dir.rs`で徹底されているパターンを新テーブルにもそのまま踏襲）ため、v1.0.0時代の1テーブルのみのDBファイルに対してv1.2で新テーブルを追加しても、初回のお気に入り登録操作時に自動生成されるだけで既存データは無傷

## 機能

### お気に入りファイル周り

#### ビューアー部

- ビューアー部での閲覧ファイルのお気に入りフラグをマーク / 解除。右クリックメニューに「お気に入り機能」ラベル下に「５つまで登録できる仮想お気に入りフォルダ」を羅列しておく、各お気に入りフォルダにチェックを付けられる、これはトグル式。１つのお気に入りフォルダに格納するだけではなく、複数の行き先と紐付け可能。その時はチェックが複数行にわたることになる

#### エクスプローラー部

- お気に入りフォルダはエクスプローラー部で管理を行う。ビューアー部での右クリックメニューに動的に名前を反映させなければいけないので、このお気に入り名前変更をトリガーに右クリックメニューリソースを整えておく必要がある
- お気に入りフォルダは右クリックメニューのトップレベルに羅列。展開式でもいいのだが、アクション数が多くなるためこれはしない
- (後回し)お気に入りタブと実フォルダのタブを左ペインのフォルダエクスプローラー上部に配置、切り替えて閲覧する。"F11" / "F12"キーでタブの切り替え
- (後回し)タブキーのキーボードコントロール実装に伴い、tabキーによるウィンドウ内部ペイン間のフォーカス切り替えをサポートする。さらにサムネ部カーソルキーとフォルダ部カーソルキーの役割の分離・実装を行う。ストレージ部はフォルダ部とほぼ同じ動作でいい。右キーでツリー展開、左で格納なわけだが、ストレージ部では左右キーは無効化していい

#### お気に入りフォルダ仕様

- 作成・削除・リネームなどはエクスプローラー部のお気に入りペインに切り替えることで可能
- "＋"のような作成ボタンは別途必要。これが押されるとお気に入りフォルダツリーの最上部に名無しのアイテムを追加する、初期からリネーム状態として配置する
- 上記仕様と合流できるようにリネームを行う、ショートカットはF2。リネームモードになった瞬間に中央に「お気に入りフォルダのリネーム」ダイアログを表示し
最上部ラベルに新しいお気に入りフォルダ名を設定してくださいと表示し
中央に名前入力エディットボックス。ここは現在値を初期値とする。新規作成なら空欄。意思決定ボタン部分はキャンセルか決定か。決定ならばその名前でリネームを確定させるが、衝突チェックは必要。衝突するのであれば、その名前はすでに使われていますを表示しダイアログに戻す。キャンセル時は何もしないが、新規作成から呼ばれている場合は新規作成アクションも白紙に戻す。また、新規作成の場合はダイアログタイトルも「お気に入りフォルダの新規作成」とする
- 名前設定時にはシンボルマーカーの設定も同時におこなう。これは後述するお気に入り列挙用マーカーとして使われる
- お気に入りフォルダ名は200文字上限とする。実在するファイルシステムとは全く別の仮想のため、ファイルシステムで使えない文字チェック等はスルーしていい
- お気に入りフォルダ数制限は0-200とする
懸念事項は後述する要望で、数多くつくっても登録メニュー側での表示が壁になる
- お気に入りペインのツリー構造はROOTから1つ下のレベルのみ（実在DIR側ツリーと違い展開・折りたたみは無い、フラット列挙）
  - 特別枠として「未整理のお気に入りフォルダ」を配置する。これはどのお気に入りフォルダにも紐付けない（テンポラリお気に入り扱いの）ファイル群を指し、ソート設定に関わらず常に最上位に固定表示する
  - お気に入りフォルダ名の左には、そのフォルダに設定されたマーカーを表示する

#### 細かい要望

- メニューから登録先のフォルダを選ぶわけだが、どうやって多数あるフォルダを選択できるようにするかが問題
  - 対応方針: ビューアーの右クリックメニューは「お気に入り詳細設定」１本に統一する。新規登録・既存の再定義いずれもここから行い、メニュー側での分岐（未登録/登録済みでの表示切り替え）は行わない
  - 詳細設定ダイアログはデュアルリストボックス方式（左＝定義済みお気に入りフォルダ一覧、右＝対象ファイルの登録先）。左右間の移動は複数選択して一括移動可能とする。フィルタ／検索欄は初期実装では見送り（後追い可能なため）
- お気に入りファイルはお気に入りフォルダに必ず格納しなければいけないわけではなく、フォルダ特有の並び替え時に先頭にスティッキー状態になるテンポラリお気に入りファイルとして存在することは可能
- お気に入りファイルは★のマーカーをサムネ左上に配置する。★の色は黄色がデフォルトだが。これはお気に入りフォルダ所属のものであれば設定したマークを左上から左下に列挙する。最大数は縦幅に依存し表示できるだけの数でいい
- 複数ファイルへの一括所属変更（既存の個別登録状態が異なる場合のマージUI）は複雑になりすぎるため、実行前確認ダイアログで「選択した◯件の既存のお気に入り設定は上書きされます」と警告した上で、選択ファイル全部を同一状態で上書きする方式とする
  - ただし複数選択・Ctrl+A全選択は現状のエクスプローラー部（`src/view_explorer.rs`）に未実装のため、本お気に入り機能のスコープには含めない。複数選択機能を土台として先に別途実装し、その上でお気に入りの一括操作に対応する

#### ネットワークファイルのお気に入り登録

- SMB等のネットワークドライブ上のファイルをお気に入り登録した後、ネットワークが未マウントの場合はリンク切れマーカーを表示する。リンク切れマーカーはサムネ右上に表示する
- ファイル単位での生死判定はタスク量が膨大になるため行わない。「SMBの接続先の大元」への到達可否を1アクションで取得し、その配下に属するファイルリンク全てのリンク切れ状態を一括で決定する
- この状態を定期ポーリングで更新することはしない（冗長なため）。状態確認（ポーリング）はリンク切れ表示中のファイルに対する明示的なオープンアクションがあった時にのみ発火する
- 到達可否判定はOSごとに手法が異なる。Windowsはネットワークドライブ経由での判定とする（UNCパスではなく割り当て済みネットワークドライブを前提）。Linux側の判定手法（マウントポイント確認等）は実装時に別途確認する

#### お気に入りマーカーの調達方針（解決済み）

- 現状 `egui 0.35`（[main.rs:114](../../src/main.rs#L114)）の`FontDefinitions::default()`に、NotoEmoji系のモノクロ記号/アイコンフォントが標準バンドルされている。新規にフォントを調達する必要はない
- カラー絵文字（フルカラービットマップ）ではなく、あえてモノクロ記号グリフを使う方針とする。モノクロ記号は任意色で着色できるため「★の色は黄色がデフォルト」のような色指定要件と両立できる（カラー絵文字は色固定のため着色不可）
- マーカーは全Unicode絵文字から自由選択にはせず、バンドル済みフォントで確実にレンダリングできることを確認した固定候補セット（20〜30種程度）に絞る。環境（Win/Linux）依存のtofu化リスクを避けるため
- お気に入りフォルダの新規作成・リネーム時のシンボルマーカー設定（既述）は、詳細設定GUI内に「マーカー選択（別ダイアログ可）」＋「着色色選択」の複合機能として実装する。組み合わせにより上限200種のお気に入りフォルダでも視覚的に区別可能な数を十分カバーできる
- 実装着手時、まず候補セットの記号がWin/Linux実機で正しく描画されるかの軽い動作確認を先行タスクとして行う
