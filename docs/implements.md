# Nekoview 実装メモ

## 概要

Rust + egui で作る Linux 専用の画像ビューアアプリ。
アーカイブ（ZIP / CBZ / 7Z）内の画像をサムネイルグリッドで一覧し、選択するとビューアウィンドウで表示する。

---

## アーキテクチャ概略

```
nekoview/
├── src/
│   ├── main.rs          # エントリポイント、引数解析、eframe 起動
│   ├── app.rs           # メインウィンドウ（egui App 実装）
│   ├── viewer.rs        # ビューアウィンドウ
│   ├── fs/
│   │   ├── dir.rs       # ディレクトリ走査（walkdir）
│   │   └── archive.rs   # ZIP / CBZ / 7Z 読み込み
│   ├── thumb/
│   │   ├── gen.rs       # サムネイル生成ロジック
│   │   └── cache.rs     # SQLite キャッシュ (index.db)
│   ├── sort.rs          # ソート軸・方向の定義と保存 (sort.toml)
│   └── config.rs        # グローバル設定（スレッド数など）
├── Cargo.toml
└── docs/
    └── implements.md    # このファイル
```

## UI レイアウト

```
┌──────────┬────────────────────────────┐
│          │  サムネイルグリッド            │
│  フォルダ  │  [img] [img] [img] ...     │
│  ツリー   │  [img] [img] [img] ...     │
│ (左ペイン) │  (右ペイン)                 │
└──────────┴────────────────────────────┘
```

- 左ペイン: エクスプローラー型フォルダツリー
- 右ペイン: アーカイブファイルのサムネイルグリッド
- ファイルをダブルクリック → 別ウィンドウでビューアを開く

---

## ソート補足

### 自然順ソート（Natural Sort）

- ZIP/CBZ 内の画像エントリ名は自然順（数値を数値として扱う順序）でソートする必要がある
- 例: `001.jpg` → `002.jpg` → `010.jpg`（辞書順だと `001` → `002` → `010` となるが、`1.jpg` → `2.jpg` → `10.jpg` は辞書順で `1` → `10` → `2` になってしまう）
- **MVP では辞書順（`sort()`）で代替**。MVP 後に `natord` クレートなどで置き換える。

---

## 対応フォーマット

- **アーカイブ**: ZIP / CBZ / 7Z（RAR・TAR は対象外）
- **画像**: `image` crate が対応するもの（JPEG, PNG, WebP, GIF, BMP など）
- 非アーカイブの生画像ファイルも閲覧できる
---

## サムネイル仕様

### 生成ロジック

1. アーカイブ内ファイルを自然順ソートで先頭から最大 5 ファイル試行
2. 各ファイルの平均輝度を計算し、白 / 黒ページを除外
3. 適切な画像が見つからなければ 5 番目（または最終ファイル）を無条件採用
4. リサイズ: 長辺 256px に収め、JPG で保存

### アスペクト比

- **1 : √2**（A4 など出版物の縦横比）でサムネイル枠を統一

### グリッド表示サイズ

- `Ctrl + ホイール` で拡縮
- **MVP**: 固定列数（4列）、サムネイル表示サイズ 256px 固定
- **本番**: ウィンドウ幅に応じた自動列数（幅 ÷ サムネイルサイズ）、表示サイズ可変設定

## .neko ディレクトリ構造

```
走査対象フォルダ/
└── .neko/
    ├── thumbs/         ← WebP サムネイル（ファイル名はアーカイブ名ベース）
    ├── sort.toml       ← ディレクトリごとのソート設定
    └── index.db        ← SQLite キャッシュ
                           （ファイルパス / 更新日時 / サムネイルパス）
```

### 読み取り専用ディレクトリへの対応

- 書き込み不可を検出したら初回のみユーザーへ問い合わせ
- 承認時: `~/.config/nekoview/ro_thumbs/` 以下に保存
- パスマッピング: 元パスを SHA256 ハッシュ化してディレクトリ名に使用
- マッピング対応表を設定ファイルに永続化

## ビューア操作

| 操作                        | 動作   |
|---------------------------|------|
| `↓` / `Space`             | 次ページ |
| `↑`                       | 前ページ |
| マウスホイール下               | 次ページ |
| マウスホイール上               | 前ページ |

- 表示モード: **1 ページ表示のみ**（シンプル優先）
- **MVP**: `egui::Window` によるフローティング表示、同時に1つのみ、画像はウィンドウにフィット（縦横比維持）
- **本番**: eframe multi-viewport でOS ネイティブ独立ウィンドウ、画像フィット/原寸を選択可能

---

## サムネイル再生成タイミング

- ディレクトリを開いたとき、`.neko/` 内のサムネイル数とアーカイブ数を比較
- 差異があれば差分のみ再生成（全件再生成しない）
- メニューの再走査を手動実行することで全件再走査して無駄なサムネイル削除と更新されたファイルのサムネイル再生成をおこなう
---

## 画像リサイズアルゴリズム

`image` crate の `FilterType` を使用。サムネイル生成とビューア表示で独立して設定できる。

### サムネイル生成

選択肢（設定値 → アルゴリズム → 特徴）：

- `nearest` — Nearest Neighbor — 最速・品質最低
- `triangle` — Bilinear — 速い・256px縮小なら品質差は小さい
- `catmullrom` — CatmullRom — バランス型
- `lanczos3` — Lanczos3 — 高品質・遅い

デフォルト: `triangle`（速度優先。256pxへの縮小では品質差が小さい）

### ビューア表示

選択肢はサムネイル生成と同じ。

デフォルト: `catmullrom`（フルサイズ表示では品質が目に見えるためバランス重視）

---

## サムネイル保存フォーマット

- `jpeg` — エンコード・デコードともに速い。非可逆・透過なし。256pxなら品質劣化は気にならない
- `webp` — エンコードがJPEGより遅いが、ファイルサイズ最小・透過対応

デフォルト: `jpeg`（生成速度がそのままサムネイル展開の体感速度に直結するため）

---

## 設定ファイル（config.toml）

```toml
[thumbnail]
format = "jpeg"              # jpeg / webp
resize_filter = "triangle"   # nearest / triangle / catmullrom / lanczos3

[viewer]
resize_filter = "catmullrom" # nearest / triangle / catmullrom / lanczos3
```

---

## バッテリー配慮

- `rayon` のスレッドプール数を設定で切り替え可能
- **デフォルト**: 省電力モード（物理コア数の半分）
- 設定: フルパワーモードも選択可

## 起動方法

```sh
# 指定パスから開始
nekoview /path/to/dir

# 引数なしはホームディレクトリ（~）から開始
nekoview
```

## 実装上の方針

- **シンプル優先**: 迷ったら機能を削る方向で判断する
- **Linux/Win対応**:方針変更
- **SMB 対応**: OS マウント前提（特別な SMB 実装は行わない）
- **Rust のみ**: Python などの他言語は使用しない
