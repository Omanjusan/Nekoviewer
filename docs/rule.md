# Nekoview MVP 開発ルール

## コスト別の進行方針

| コスト | 対応 |
|--------|------|
| 低     | 実装後、確認事項をまとめて提示 → ユーザー承認後に次へ進む |
| 中〜高 | 着手前にテスト項目をユーザーに質問して確定（デバッグ主体開発）|

## 共通ルール

- ステップをまたいで確認なしに次へ進まない
- コスト中〜高のステップはさらに細分化してよい。細分化した項目ごとにも確認タイミングを入れる
- 1000トークン超の検討が必要と判断したときは途中で停止してユーザーに問い合わせる
- 各ステップで実装した内容は下記「ステップ実装ログ」に随時記録し、チャットをまたいでも参照できるようにする

### キャッシュ機構（ステップ7d）
- ステータス: 実装中
- 細分化: 7d-a（PageCache構造体）→ 7d-b（ワーカースレッド）→ 7d-c（NekoviewApp統合）→ 7d-d（ViewerState GPU管理改修）

#### 仕様
- CPUキャッシュ: `RgbaImage` を 500MB 上限でスライディングウィンドウ管理（後方10・前方5）
- GPUキャッシュ: ウィンドウ内ページ分の `TextureHandle` のみ保持
- ワーカースレッド: `std::thread` + `mpsc::channel`
- ロード中表示: グレー
- ビューアを閉じてもCPUキャッシュ保持（`NekoviewApp` が管理）
- eviction: 予算内なら残す、超過時は現在ページから最遠のページを解放
